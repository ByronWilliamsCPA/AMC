> **Purpose:** Define how Phase 3 deliverable 3.3 ("Responsive/mobile layout;
> keyboard navigation; basic a11y pass") is delivered: build a full-stack
> Playwright end-to-end harness as the measuring instrument, use it to audit
> the SPA, fix the gaps it surfaces, lock each fix with a regression assertion,
> and document the exam-page Lighthouse result.

## 1. Goal and acceptance mapping

Source of truth: [PROJECT-PLAN.md](../../planning/PROJECT-PLAN.md) Phase 3.

| Acceptance criterion (plan) | How this design satisfies it |
|---|---|
| 3.3 Responsive/mobile layout | `responsive.spec.ts` asserts no horizontal overflow and reachable controls at 320 to 1280px; CSS fixes close gaps |
| 3.3 Keyboard navigation | `keyboard.spec.ts` drives skip-link, palette roving tabindex, dialog focus trap, and a keyboard-only exam submission |
| 3.3 Basic a11y pass | `a11y.spec.ts` runs `@axe-core/playwright` (WCAG 2.0/2.1 A+AA) on every key page in light and dark |
| Lighthouse: exam page interactive < 2s | `lighthouse.spec.ts` runs `playwright-lighthouse` on the exam runner and asserts interactive < 2s; measured number recorded in the report |
| All tests pass | New e2e suite plus existing vitest and pytest suites green; CI job added |

Depth (decided): **audit + fix the real gaps + regression tests**, not audit-only
and not full VPAT-style AA conformance. The SPA already has strong a11y
foundations (native controls, skip link, focus management, roving-tabindex
palette, live regions, `vitest-axe` smoke tests, `eslint-plugin-jsx-a11y`,
44px touch targets, `prefers-reduced-motion`); 3.3 verifies behaviour at the
edges jsdom cannot see and fixes what fails.

## 2. Architecture decision: full-stack e2e on SQLite + synthetic seed

Decided approach (Approach 1 of three considered):

- The e2e suite runs the **real** FastAPI app, real auth, and real exam runner.
- Database is a throwaway on-disk SQLite file, mirroring the existing
  in-memory-SQLite test pattern. No Docker/Postgres in CI.
- Data comes from a committed **synthetic** `content/e2e_seed.json`, never the
  real `content/amc_data.json`.

### Why not the alternatives

- **Postgres via docker-compose:** prod-parity DB engine, but slower CI, more
  flake surface, and zero benefit for a11y/layout/Lighthouse, which is what 3.3
  measures.
- **Real `amc_data.json` in CI:** would place the private AMC scans into CI
  infrastructure, violating the carried-forward rule that the scans never leave
  the private host. One synthetic 25-problem exam exercises every responsive and
  a11y code path the palette and runner have.

### Data-privacy constraint (#CRITICAL)

```text
# #CRITICAL: data-integrity/security: content/amc_data.json (real AMC scans) is
#   gitignored and MUST NOT enter the public repo or CI.
# #VERIFY: e2e seed uses only the synthetic content/e2e_seed.json; confirm
#   e2e_seed.json contains no real problem text/images before commit.
```

## 3. Components

### 3.1 Synthetic seed fixture: `content/e2e_seed.json`

Matches the loader schema in [src/amc/seed.py](../../../src/amc/seed.py):

```jsonc
{
  "tests": {
    "E2E-EXAM-1": {
      "contest": "AMC E2E", "year": 2099, "exam": "",
      "durationSec": 4500, "scoreMode": "<match seed contract>",
      "answers": ["A","B","C","D","E", ...25 entries...],
      "mode": "latex",
      "problems": [
        {"n": 1, "q": "<latex>", "choices": ["..."], "sol": "https://example.test/1"},
        // ...25 problems: mostly A-E text/LaTeX choices, one LaTeX-heavy
        //    problem to exercise KaTeX overflow, and (if image serving is
        //    wired) one image problem to exercise the image render path.
      ]
    }
  },
  "instruments": { "e2e-diag": { "sections": [ { "items": [ /* 1 auto + 1 manual */ ] } ] } },
  "catalog": [ /* minimal entry so the diagnostic is reachable */ ]
}
```

Requirements:

- Exactly 25 problems so the question palette renders at full size (worst case
  for the 5-column grid at 320px).
- Validate against [content/validate_content.py](../../../content/validate_content.py)
  before commit.
- `scoreMode`, instrument, and catalog shapes copied from the real contract;
  exact field values resolved during planning by reading `seed.py` and the
  models.

```text
# #ASSUME: data-integrity: problem images are served by the app from a path the
#   synthetic img value can reference.
# #VERIFY: during planning, locate the static/image mount; if image serving is
#   not trivially available in e2e, omit the image problem and note it as a
#   covered-elsewhere gap rather than fabricate a broken image (which would
#   itself be an a11y finding).
```

### 3.2 Harness config: `frontend/playwright.config.ts`

- `use.baseURL = http://localhost:3000`.
- Two `webServer` entries (Playwright waits on each before running specs):
  1. **API**: `scripts/e2e/run-api.sh` (new) which:
     - exports `DATABASE_URL=sqlite+aiosqlite:///./.e2e/amc_e2e.db` (gitignored
       `.e2e/` dir, recreated each run),
     - `uv run alembic upgrade head`,
     - `uv run python -m amc.seed --amc content/e2e_seed.json --diag content/e2e_seed.json`,
     - bootstraps the test users (see 3.3),
     - execs `uv run uvicorn amc.main:app --port 8000`;
     Playwright `url: http://localhost:8000/health/live`.
  2. **Frontend**: `npm run build && npm run preview -- --port 3000`;
     Playwright `url: http://localhost:3000`.
- `projects`: `chromium-desktop` (1280x800), `webkit-mobile` device profiles for
  the viewport set; Lighthouse spec restricted to a Chromium project.
- Reporters: `list` + `html`; `trace: 'on-first-retry'`, `screenshot:
  'only-on-failure'`.

### 3.3 Required config change: `vite preview` proxy

`frontend/vite.config.ts` defines `server.proxy` (dev) but no `preview.proxy`.
The production-build preview used by e2e must forward `/api` and `/health` to
`:8000` for the same-origin `amc_session` cookie to work. Add a `preview.proxy`
block mirroring `server.proxy` (same `VITE_DEV_API_TARGET` default).

### 3.4 Auth bootstrap and test users (`globalSetup`)

`create_admin` only mints staff roles (`_STAFF_ROLES` = admin, coach); students
self-serve via invite. To drive the real student exam path without a test-only
auth backdoor:

1. Create a coach out-of-band: `uv run python -m amc.create_admin --role coach
   --email coach@example.test` (password via `AMC_ADMIN_PASSWORD`).
2. In `frontend/e2e/global-setup.ts`: log in as the coach via
   `POST /api/v1/auth/login`, mint a student invite via the invite API, then
   register the student via `POST /api/v1/auth/register` with that invite.
3. Log in as the student, save `storageState` to `frontend/e2e/.auth/student.json`;
   specs load it so they start authenticated.

```text
# #ASSUME: security: an invite-mint endpoint exists and is callable by a coach
#   (InvitePage.tsx implies an invite API).
# #VERIFY: during planning, read src/amc/api for the invite route and its
#   request/response shape; if absent, fall back to driving exam specs as a
#   coach account and note the student-onboarding path as separately covered.
```

### 3.5 Audit specs (the measuring instrument)

Key pages under test: login, exam list, **exam runner mid-exam**, exam review,
diagnostic runner, progress.

- `frontend/e2e/a11y.spec.ts` - `@axe-core/playwright`, tags
  `wcag2a wcag2aa wcag21a wcag21aa` (matching the existing
  `src/components/ui/a11y.test.tsx`), zero violations per page, run in `light`
  and `dark` `colorScheme`.
- `frontend/e2e/responsive.spec.ts` - at 320x568, 375x667, 667x375 (landscape),
  768x1024, 1280x800: assert `document.scrollWidth <= clientWidth + 1` (no
  horizontal overflow) on each page; assert the palette drawer toggle works
  below 720px; assert primary controls are >= 44px; capture screenshots as
  artifacts (not pixel-asserted, to avoid flake).
- `frontend/e2e/keyboard.spec.ts` - skip-link reaches main; nav order; palette
  roving tabindex (Arrow/Home/End); dialog focus trap + Esc; complete an exam
  submission keyboard-only.
- `frontend/e2e/lighthouse.spec.ts` (Chromium only) - `playwright-lighthouse` on
  the exam runner; assert time-to-interactive < 2000ms and accessibility
  category >= 95; record measured numbers (note ambient variance).

### 3.6 Fixes (audit-driven)

The harness surfaces the authoritative list. From the static audit, the
expected fixes are:

- **Palette grid** (`features/exam/Palette.module.css`): 5 fixed columns can
  overflow at 320px; move to responsive columns (e.g. `repeat(auto-fit,
  minmax(...))` or fewer columns <=360px).
- **Dark mode** (`styles/tokens.css`): dark token values exist but a
  `@media (prefers-color-scheme: dark)` switch may be missing; wire it and fix
  any contrast violations axe reports.
- **Exam runner** (`pages/ExamRunnerPage.module.css`): header wrap at mid
  widths; drawer/dialog content reachability in landscape (`max-height` /
  overflow).
- **Question render** (`features/exam/Question.*`): equation/image overflow at
  320 to 375px (`overflow-x` containers).

Each fix lands with the spec assertion that now passes. Unknown until the suite
runs first; the workflow is run -> triage -> fix -> green.

### 3.7 Documentation

- Report at `docs/accessibility/3.3-a11y-mobile-2026-06.md`: scope, WCAG target
  (2.1 AA), browser/viewport matrix, per-page axe results, responsive results,
  keyboard results, measured Lighthouse numbers on the exam page, the list of
  fixes made, residual/known limitations (e.g. i18n out of scope), and how to
  run the suite locally.
- `CHANGELOG.md` entry; a "running e2e tests" note in the frontend README or
  docs.

### 3.8 CI

- New job `frontend-e2e` (in `.github/workflows/ci.yml` or a new
  `frontend-e2e.yml`): setup uv + Python 3.12, setup Node, `uv sync`,
  `npm ci` in `frontend/`, `npx playwright install --with-deps chromium webkit`,
  run `npm run test:e2e`. Upload the Playwright HTML report and traces as
  artifacts on failure. Lighthouse spec gated to Chromium.
- New frontend dev-deps: `@playwright/test`, `@axe-core/playwright`,
  `playwright-lighthouse`, `lighthouse`. New `test:e2e` npm script.

## 4. Scope guards (YAGNI)

- No pixel-regression snapshots in CI (screenshots are artifacts only).
- No i18n work (flagged as future).
- No deep staff/coach e2e beyond the exam-taking path the criterion names.
- Firefox off by default (Chromium + WebKit cover the engines that matter).

## 5. Delivery

- Branch `test/3.3-a11y-mobile`; signed conventional commits typed per change
  (`test:`, `fix(frontend):`, `build(frontend):`, `ci:`).
- Gates before PR: `npm run test` (vitest), `npm run test:e2e` (Playwright),
  `uv run pytest -q`, `pre-commit run --all-files`.

## 6. Open assumptions to resolve during planning

1. Image serving path for the synthetic image problem (3.1).
2. Invite-mint endpoint shape for the student-registration bootstrap (3.4).
3. Exact `scoreMode` / instrument / catalog field values for the seed contract.
4. Whether `vite preview` honours a `preview.proxy` block on the installed Vite
   version (6.0.6); if not, run the API on the same origin via a different
